# ログイントラブルシューティング

## レートリミットに引っかかった場合

### レートリミットの症状

ログインボタンを押すと「サーバーのアクセス制限に達しました。しばらく待ってから再試行してください」と表示される。

### レートリミットの原因

Mastodon サーバーは `POST /api/v1/apps`（アプリ登録）にレートリミットを設けている。capsicum は初回ログイン時にこのエンドポイントを呼ぶため、短時間に複数回ログインを試みるとブロックされる。

同じホストに既存アカウントがある場合はアプリ登録をスキップするが、全アカウントを削除した後の再ログインではスキップできない。

### 対処手順

1. **5分程度待つ** — Mastodon のデフォルトレートリミットは 5 分間隔でリセットされる
2. 待ってから再度「ブラウザでログイン」を押す
3. それでも解消しない場合は、サーバー側のレートリミット設定を確認する

### サーバー管理者向け: レートリミットの確認

Mastodon のレートリミットは Rack::Attack（Rails ミドルウェア）で実装されており、`/api/v1/apps` のリミットは厳しめに設定されている。開発・テスト中に繰り返しログインすると容易に到達する。

- Rack::Attack の設定は `config/initializers/rack_attack.rb` にある
- `throttle('throttle_api_sign_up')` 等が `/api/v1/apps` に適用される
- Redis に保存されたカウンタを `rails console` でリセットすることも可能

ただし、サーバー側のリミット緩和は対症療法に過ぎない。capsicum 側で不要な `/api/v1/apps` 呼び出しを減らすのが正しい対策（後述）。

### レートリミットの根本対策（将来の改善）

- clientSecret をホスト単位で永続化し、アカウント削除後もアプリ登録をスキップできるようにする（Issue 化検討）

## ブラウザからアプリに戻れない場合

### リダイレクト失敗の症状

ログイン時にブラウザが開き、サーバー側で認可を承認した後、capsicum アプリに自動で戻らない。ブラウザに `capsicum://oauth?code=...` のような URL が表示されたまま止まる。

### リダイレクト失敗の原因

capsicum は `capsicum://` カスタム URL スキームでブラウザからのリダイレクトを受け取る。以下の状況でリダイレクトが失敗することがある:

- **Android 12 以降**: Custom Tabs の仕様変更により、カスタムスキームへのリダイレクトが安定しない場合がある
- **デフォルトブラウザの設定**: 一部のブラウザがカスタムスキームを処理しない
- **iOS**: 通常は ASWebAuthenticationSession が処理するため発生しにくいが、ゼロではない

### リダイレクト失敗時の対処手順

#### Misskey サーバーの場合

MiAuth フォールバックが自動で働く。ブラウザで認可を承認した後、アプリに戻れなくても、手動でアプリに切り替えればログインが完了する。

#### Mastodon サーバーの場合

リダイレクトが失敗すると、認証コードの手動入力ダイアログが表示される。ブラウザのアドレスバーに表示された `capsicum://oauth?code=XXXXX` の URL（または `code=` 以降のコード部分）を貼り付けてログインを完了できる。

### 過去に発生した問題: ブラウザセッションの残留

以前は `preferEphemeral: false`（ブラウザセッション共有）を使用しており、OAuth 認可後にブラウザにセッションが残り、次回ログイン時に Mastodon WebUI に遷移してしまう問題があった。アプリのアンインストール以外に解消方法がなかった。

現在は `preferEphemeral: true`（使い捨てセッション）に変更済みのため、この問題は発生しない。代わりに毎回サーバーのパスワード入力が必要になる。
